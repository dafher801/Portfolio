계획 수립의 자동화 Automated Planning

포커에서는 프로젝트 계획 수립 자체를 AI에게 위임하였다. 계획을 마일스톤 → 에픽 → 태스크의 계층 구조로 분리하고, 각 계층에 규칙을 부여하였다. 모든 과정에서 Self-Reflection 기법을 적용하여, 규칙 준수 여부를 스스로 검증하도록 설계하였다.

◈ 계획 단위별 규칙 구조
Milestone
검증 체크포인트
빌드를 통해 목표 달성이 가능해야 하며, 후행 마일스톤이 선행 구현을 수정하지 않도록 의존 순서대로 정렬한다.
Epic
독립된 기능
에픽 하나가 완료되면 해당 기능이 동작하는 상태가 되어야 한다. 마일스톤에 소속되며, 하나의 마일스톤은 하나 이상의 에픽으로 구성된다.
Task
단일 작업 단위
원칙적으로 하나의 소스 코드 파일에 대응되며, 설명만으로 작업 수행이 가능할 만큼 명확해야 한다. 소스 코드의 설계 규칙은 주어진 레이어 구조 원칙을 따른다.
◈ 레이어 참조 흐름
ViewUI·조작
Director흐름 제어
Usecase게임 로직
Gateway입출력
Entity데이터·규칙

각 계층은 아래 계층만 참조하며, 같은 레이어 간 참조는 금지한다. 이 구조를 통해 AI가 하나의 태스크를 수행할 때 참조해야 할 코드 범위를 명확히 한정하여, 컨텍스트 윈도우를 효율적으로 활용할 수 있었다.

실험 결과Results

✓ 컴파일 에러, 테스트 실패 등 로그로 직접 전달 가능한 에러는 AI가 자체적으로 해결할 수 있었다.
✗ 인게임 플레이에서 감지되는 버그는 로그로 전달 가능한 에러보다 해결 정확도가 떨어졌다.
→ 버그의 예상 위치를 구체적으로 짚어줄수록 해결 정확도가 높아졌다.
✓ 씬과 프리팹의 계층 구조 구성은 AI가 안정적으로 수행하였다.
✗ UI/UX 미세 조정은 C# 코드 작성만으로는 자동화가 어려웠다.
→ HTML → Figma → Unity GUI / UXML 파이프라인을 통해 자동화가 가능할 것으로 보이며, 추가 실험이 필요할 것으로 판단된다.
✓ AI가 작성한 코드가 게임 상에서 대부분 잘 작동하였다.
✗ 소스 코드 작성 시, 레이어 간 의존 방향 규칙을 무시하고 코드를 생성하는 경우가 발견되었다.
→ 정해진 계층 외의 참조를 물리적으로 차단하여 규칙 위반을 방지할 수 있을 것으로 보이며, 추가 실험이 필요할 것으로 판단된다.
