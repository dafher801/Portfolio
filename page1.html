커버 레터 Cover Letter

1 매 회차마다 시나리오가 바뀌는 RPG

매 회차마다 새로운 시나리오로 진행되는 RPG를 만드는 연구를 진행하였고, 이를 위한 자동화 파이프라인을 설계했습니다. 이 연구는 시나리오 생성 → 월드 구성 → 시스템 연동의 세 단계로 진행되었습니다.

① 시나리오 생성 — 시나리오 작법에 대한 도메인을 프롬프트로 구조화하고, 매 회차마다 새로운 시나리오를 생성하도록 설계했습니다.

② 월드 구성 — 생성된 시나리오를 프롬프트 체인과 Self-Reflection을 통해 플레이 가능한 게임 월드로 변환했습니다. LLM이 스스로 결과를 검증·수정하여 품질을 관리합니다.

③ 시스템 연동 — 대화 유형 자동 분류 체계를 통해, LLM 출력이 퀘스트·전투 등의 이벤트로 자동 변환되도록 하네스를 구현했습니다.

이를 통해 문제 정의 → 도메인 파악 → 프롬프트 구조화 → 자체 품질 관리 → 하네스 설계라는 자동화 설계의 사이클을 경험했습니다. 이는 현장의 업무를 파악하고 그에 맞는 에이전트를 만들어주는 FDE의 업무와 본질적으로 같은 구조라고 생각합니다.

2 바이브 코딩, 어디까지 가능한가?

바이브 코딩의 한계를 시험하기 위해 체스와 포커를 각각 3일동안 구현했습니다. 체스에서는 수동으로 계획을 수립하고 단계별 구현을 지시하여 멀티플레이와 고수준 AI까지 동작하는 결과물을 얻었고, 포커에서는 이를 더 발전시켰습니다.

① 계획 자동화 — 프로젝트 계획을 마일스톤 → 에픽 → 태스크 계층으로 분리하고, 계획 수립 과정을 Self-Reflection 프롬프트로 AI로 자동화했습니다.

② 레이어 구조 — View → Director → Usecase → Gateway → Entity의 단방향 구조를 적용하여, 태스크당 참조 범위를 한정하고 컨텍스트 윈도우를 효율적으로 활용했습니다.

③ 버그 해결 — 인게임에서 감지되는 버그는 AI가 자체적으로 해결하지 못하는 문제가 발생했으나, 레이어 구조를 기반으로 버그의 예상 위치를 구체적으로 짚어주자 해결 정확도가 높아지는 것을 확인했습니다.

이를 통해 문제 정의 → 도메인 파악 → 프롬프트 구조화 → 자체 품질 관리 → 하네스 설계의 사이클이 다른 분야에서도 적용되는 것을 확인했을 뿐만 아니라, 각각 3일이라는 짧은 기간 안에 작동하는 결과를 만들어내며 빠른 프로토타이핑이 가능함을 확인했습니다. 이는 클라이언트에게 빠르게 프로토타입을 제시하고 피드백을 반복하는 업무를 수행하는 데 직접적으로 도움이 될 것입니다.

★ 파이프라인 설계, 코딩 에이전트 활용 및 AI Native 개발 경험을 갖춘 인재로써, 현장의 업무를 AI로 전환하는 FDE의 핵심 업무에 즉시 기여할 수 있다고 확신합니다.
